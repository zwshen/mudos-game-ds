============    圖形模式    ============ 
======================================== 
void set_color_depth(int depth); 
   設置點的格式來由隨後調用 set_gfx_mode() 和 create_bitmap() 時 
   使用. 有效的顏色深度有 8 (缺省值), 15, 16, 24, 和 32 位. 
 
int set_gfx_mode(int card, int w, int h, int v_w, int v_h); 
   切換圖形模式. 參數 card 必須是以下值中的一個: 
 
      GFX_TEXT          - 返回文本模式 
      GFX_AUTODETECT    - 讓 Allegro 選擇一個合適的圖形驅動程序 
      GFX_VGA           - 選擇 VGA 模式 13h (320x200 或 320x100) 
      GFX_MODEX         - 選擇平面扭轉的(plannar tweaked) VGA 模式 
      GFX_VESA1         - 使用 VESA 1.x 驅動程序 
      GFX_VESA2B        - 使用 VBE 2.0 分頁模式驅動程序 
      GFX_VESA2L        - 使用 VBE 2.0 線形頁面緩衝驅動程序 
      GFX_VBEAF         - 使用 VBE/AF 加速器 API (未完成) 
      GFX_XTENDED       - 使用斷鏈式(unchained) 640x400 模式驅動程序 
      GFX_ATI           - 使用 ATI 18800/28800 驅動程序 
      GFX_MACH64        - 使用 ATI mach64 驅動程序 
      GFX_CIRRUS64      - 使用 Cirrus 64xx 驅動程序 
      GFX_CIRRUS54      - 使用 Cirrus 54xx 驅動程序 
      GFX_PARADISE      - 使用 Paradise 驅動程序 
      GFX_S3            - 使用 S3 驅動程序 
      GFX_TRIDENT       - 使用 Trident 驅動程序 
      GFX_ET3000        - 使用 Tseng ET3000 驅動程序 
      GFX_ET4000        - 使用 Tseng ET4000 驅動程序 
      GFX_VIDEO7        - 使用 Video-7 驅動程序 
   參數 w 和 h 指定了你想要的屏幕分辨率. 可能的模式有: 
 
        VGA 13h 模式, 大小為 320x200 (256 色), 使用 GFX_VGA 驅 
        動程序.這個在任何 VGA 下都能工作, 但是不支持大的虛擬屏 
        幕和硬件滾屏. 這個驅動程序還提供了雙線 320x100 模式,它 
        能在整個 320x200 的虛擬屏幕中滾動.  
        Mode-X, 使用 GFX_MODEX 驅動程序. 這個能夠在任何 VGA 卡 
        上工作, 提供了扭轉的(tweaked) 分辨率和 256 種不同的顏色. 
        穩定的 mode-X 分辨率: 
           正方形外觀的比例: 320x240 
           長方形外觀的比例: 256x224, 256x240, 320x200, 320x400, 
                             320x480, 320x600, 360x200, 360x240, 
                             360x360, 360x400, 360x480 
           這些我在所有的顯卡的顯示器上都測試通過了. 
        不穩定的 mode-X 分辨率: 
           正方形外觀的比例: 360x270, 376x282, 400x300 
           長方形外觀的比例: 256x200, 256x256, 360x600, 376x308, 
                             376x564, 400x150, 400x600 
           這些只能在部分顯示器上工作. 它們可以在我的老 
           機器上工作, 但是在我的新顯示器上工作的不那麼好. 如果 
           你擔心使用這些模式會對你的顯示器可能造成傷害, 那麼大 
           可不必. 當然我不能向這個提供任何的擔保, 也許你的硬件 
           還是燒掉了,但是我相信這一系列 tweak 模式不會造成任何 
           危險. Robert Schmidt 的 TWEAK 程序文本裡寫道: 
              "一些時間以前, 向顯卡寄存器裡置一些非法或是不被支 
              持的值或組合可能對你的顯示器和你的健康都是一種冒 
              險. 我 *從來沒有* 聲明如果你使用 TWEAK 不會有什麼 
              壞的事情發生,然而我很樂意相信不會有. 我從來沒有聽 
              說在試著使用 TWEAK 以外或一般的 VGA 扭轉(tweaking)  
              時有什麼危險的事情發生." 
        因為複雜的分頁位圖組織方式, 所有的 mode-x 繪圖函數都比  
        13h 模式慢, 但是整塊的區域填充和平面調整方式的從顯存的 
        一部分到另一部分的移動能夠更快點, 尤其是在老的硬件上. 
        Mode-X 能尋址 VGA RAM 的整個 256k 空間, 因此硬件滾屏和 
        多頁面轉換就辦的到了, 它也可以分割開屏幕來實現上部顯示 
        部分的滾動,而下部分卻靜止用來顯示一個狀態條.   
        基本 SVGA 模式, 640x480, 800x600, 和 1024x768. 這些能 
        夠在任何 SVGA 卡上使用: 如果不能用, 就搞套 UniVBE 看看 
        能否用了. 顏色的深度依賴於你的硬件. 所有的卡都支持 15  
        和 16 位的顏色, 但是我建議你最好是兩個都去支持 (這並不 
        難...) 因為有可能有一種可能不能用. 一些卡對 24 和 32  
        位真彩色都提供了, 這樣你就有 24 (節約內存) 和 32 (更快 
        一些) 兩種選擇, 但是許多老卡沒有 32 位模式而一些新卡卻 
        不支持 24 位模式.使用 VESAINFO 測試程序來看看你的 VESA 
        驅動程序提供了哪些模式. 許多顯卡還支持 640x400, 
        1280x1024, 和 1600x1200, 但是這些不是在任何地方都能用 
        的, 比如 S3 芯片就沒有 640x400 模式. 其它的奇異的分辨 
        率也有, 比如, 一些 Tseng 板可以實現 540x350, 還有 
        Avance Logic 有 512x512 模式. 
        Xtended 模式, 由 PCGPE 的 Mark Feldman 描述, 大小是 
        640x400 (256 色) 使用 GFX_XTENDED 驅動程序. 這個使用 
        VESA 來選擇 SVGA 模式 (所以它只能在支持 VESA 640x400 
        分辨率的卡上工作), 然後像 mode-X 那樣將 VGA 硬件做斷鏈 
        接處理 (unchain). 這使得整個屏幕不需要頁切換尋址, 但是 
        硬件滾屏和多頁面轉換辦不到. 這個驅動程序永遠不會被自動 
        檢測到 (取而代之的是正常的 VESA 640x400 模式),因此如果 
        你想使用它就必須明確的向 set_gfx_mode() 傳入參數 
        GFX_XTENDED. 
        UniVBE 提供了一個可滾屏的低分辨率不同顏色深度的模式 
        (320x200, 320x240, 320x400, 320x480, 360x200, 360x240, 
        360x400, 和 360x480 都可以在我的 ET4000 上以每點 8,15, 
        或 16 位 工作). 這些很可愛, 讓我們不必使用複雜的mode-X 
        平面設置就可以實現滾動和多頁面切換. 可惜的是它們不是基 
        本模式, 所以你需要 UniVBE 來使用它們. 
        VBE/AF 是 VBE 2.0 基本模式的超集, 它提供了一個 API 來 
        利用硬件加速特性. VBE/AF 現在只能被 UniVBE 支持, 而且 
        SciTech 現在只在 Mach64 芯片上實現這個功能. Allegro盡 
        可能的支持我在自己的機器上測試了的基本部分,  即沒有加 
        速的繪圖模式. 我真的很想讓這個驅動程序支持加速函數,但 
        是只有等 SciTech 在 Matrox Mystique 上加入這個工具了, 
        或者有人為我買塊 Mach64 (建議 建議 :-) 
   參數 v_w 和 v_h 指定了最小虛擬屏幕的大小, 因為你需要一個大 
   的虛擬屏幕來做硬件滾屏或多頁面切換. 如果你不在乎虛擬屏幕的 
   大小就應該將它們設為零. 虛擬屏幕回造成許多混亂, 但是它們真 
   的很簡單. 留神: 技術說明來了, 所以你可以跳過這一段的剩下部 
   分 :-) 將顯示內存想像成一張矩形的紙, 通過一個上面有一個小 
   洞(你的顯示器)的卡片來看它. 既然紙比洞要大, 你在任何時候就 
   只能看見紙的一部分, 但是移動卡片你就能改變看到的景象. 你能 
   將洞放在一個地點而副略顯示內存其它看不見的部分, 但是你可以 
   利用移動屏幕窗口來實現許多有用的效果, 或者可以在顯存隱藏的 
   部分先畫好圖片然後切換過去來顯示它們. 舉個例子, 你選擇了一 
   個虛擬屏幕為 1024x1024 的 640x480 模式, 然後可以在這個大區 
   域上移動可視的部分. 初始化時, 可視部分在顯存的左上角, 這個 
   設置看起來像這樣: 
      (0,0)------------(640,0)----(1024,0) 
        |                  |           | 
        |   可視屏幕部分   |           | 
        |                  |           | 
      (0,480)----------(640,480)       | 
        |                              | 
        |        顯存的其它部分        | 
        |                              | 
      (0,1024)--------------------(1024,1024) 
   這是什麼? 你正在用相稱的字體來顯示? 呵呵. 
 
   當你調用 set_gfx_mode() 時, v_w 和 v_h 參數提供你的程序能 
   夠接受的最小虛擬屏幕的大小.大小的範圍通常很受限制,Allegro 
   一般最終會創建一個比你要求要大的多的虛擬屏幕.在 SVGA 卡上 
   一兆的顯存能夠讓你創建 1024x1024 的虛擬屏幕 (256 色) 或是 
   1024x512 (15 或 16 bpp), 而 512k 顯存你可以得到 1024x512  
   (256 色). 其它的大小可能或不可能得到: 不要假定它們能工作. 
   在 mode-X 下虛擬屏幕的寬度可以是物理屏幕寬度的 8X 倍或是 
   與之相同, 於是模擬高度就被相應的設置了 ( VGA 有 256K 顯存,  
   所以虛擬高度將是 256*1024/虛擬寬度). 在你選擇一個圖形模式 
   以後, 物理的和虛擬的屏幕大小可以用宏 SCREEN_W(), 
   SCREEN_H(), VIRTUAL_W(),和 VIRTUAL_H()來測得.如果 Allegro 
   不能選擇一個合適的模式, set_gfx_mode() 將返回一個負數並且 
   將問題的描述儲存在 allegro_error 裡. 否則它返回零. 
 
int scroll_screen(int x, int y); 
   嘗試硬件滾動屏幕來顯示虛擬屏幕的不同部分 (初始化時位置在 
   0,0, 就是左上角). 成功則返回零: 如果圖形驅動程序不能做硬 
   件滾屏或者虛擬屏幕不夠大則它可能出錯. 如果圖形卡需要這個 
   函數將等待垂直回掃, 因此你不必自己調用 vsync(). 你可以使 
   用這個來把你的屏幕在大的虛擬屏幕中移動, 或者在虛擬屏幕的 
   兩個不重疊的部分間進行頁切換. 注意在屏幕位圖上超出原始坐 
   標作畫時, 你必須改變剪裁矩形: 見後. Mode-X 滾屏是可信的, 
   它可以在任何卡上工作. 可惜許多 VESA 工具 (包括 UniVBE!) 
   只能處理水平上的 4 點遞增滾動, 所以平滑的水平滾動是不可 
   能的. 這真是慚愧, 但是我沒有辦法解決它. 大多數 VESA 工具 
   似乎在真彩模式下的滾動都有非常多的 bug, 因此我建議你不要 
   依賴這個例程來正確的進行真彩模式下的滾屏,除非你確認  
   UniVBE 已經加載了.Allegro 將在滾動屏幕時進行必要的垂直回 
   掃同步, 因此你不必在條用前面調用 vsync(). 這意味著 
   scroll_screen() 和 vsync() 造成的延時時間相同. 
 
void request_modex_scroll(int x, int y); 
   這個函數只能在 mode-X 下使用, 垂直回掃中斷模擬器要被激活. 
   它請求硬件滾屏到指定位置, 但是馬上返回而不是等待一次回掃. 
   這個滾動將在下次垂直回掃中被實現. 這個函數作三緩衝時很有 
   用 (見 examples/ex20.c). 
 
int poll_modex_scroll(); 
   這個函數只能在垂直回掃中斷模擬器被激活時使用在 mode-X  
   上. 如果硬件滾屏在之前被 request_modex_scroll() 設置而還 
   在等待實現它將返回非零值如果已經滾動已經發生則返回零. 
 
void split_modex_screen(int line); 
   這個函數只在 mode-X 下有效. 它將在指定行將 VGA 的顯示分 
   成兩個部分. 屏幕上面的部分可以利用 scroll_screen() 函數 
   被滾動到顯存的任意部分,而指定行下面的部分將被保留並且重 
   屏幕位圖的坐標 (0,0) 開始顯示. 在分割屏幕以後你一般想滾 
   動所以上面的部分是從接下來的顯存開始向顯存後面延伸,然後 
   創建兩個子位圖來控制這兩個部分 (見 examples/ex19.c 看看 
   這些是怎樣運作的演示). 關掉分割用 split_modex_screen(0).
